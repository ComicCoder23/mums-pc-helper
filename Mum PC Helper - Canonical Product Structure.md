MUM PC HELPER — CANONICAL PRODUCT STRUCTURE


Purpose
This document resets Mum PC Helper into the correct structure so Google Drive, GitHub, and Notion stay tidy.


Main rule
One function = one canonical home.
No more loose doc sprawl.
No more mixing build docs with customer-facing product files.


MASTER STRUCTURE


1. BUILD
Use for internal creation, planning, editing, testing, and support material.
This is for you, not the end user.


Contents:
- script drafts
- internal notes
- interpretation guides
- decision trees
- sprint docs
- session/support notes
- future ideas


2. PRODUCT
Use for the actual downloadable product package.
This is what Mum or any end user should receive.


Contents:
- Start Here
- User Guide
- Quick Run Card
- One Pager
- script files
- safe packaged reports/templates
- version notes if needed


3. ARCHIVE
Use for anything replaced, merged, experimental, or redundant.
Nothing deleted blindly.
Anything uncertain goes here first.


Contents:
- duplicate docs
- superseded guides
- merged drafts
- abandoned pack variations


HOW TO CLASSIFY THE CURRENT FILES


PRODUCT CANDIDATES
These are likely user-facing:
- Mum PC Helper Pack — Start Here
- Mum PC Helper - Quick Run Card
- Mum PC Helper - Visit Workflow One Pager
- Mum PC Helper - Action Decision Tree
- PowerShell script docs once converted into actual .ps1 files and simple instructions


BUILD CANDIDATES
These are likely internal:
- Mum PC Helper - Session Notes
- Mum PC Helper - Safe Tools List
- Mum PC Helper - Safe Scripts Plan
- Mum PC Helper - Results Interpretation Guide
- Mum PC Helper - Run Order for Scripts
- Mum PC Helper - Browser Triage Guide
- Mum PC Helper - First Visit Checklist
- Mum PC Helper - Cleanup Order
- Mum PC Helper - What To Ask Mum
- Mum PC Helper - Master Index


ARCHIVE CANDIDATES
Anything overlapping after the merge pass should move here.
Examples:
- duplicate process docs
- docs whose content has been absorbed into Start Here, Quick Run Card, or One Pager


PRODUCT PACK TARGET
The final downloadable product should probably be reduced to:
1. Start Here
2. Quick Run Card
3. Visit Workflow One Pager
4. User Guide
5. Script Bundle
6. Optional Safety Notes


That is a product.
Everything else is build/support material.


GITHUB / DRIVE / NOTION ROLES


GitHub
- canonical build assets
- script source files
- version control
- product packaging files


Google Drive
- distribution assets
- customer-facing downloadable pack
- working docs when needed


Notion
- control tower
- decisions
- structure
- classification rules
- roadmap and release tracking


NEXT ACTIONS
1. Stop creating new loose docs.
2. Convert current docs into Build / Product / Archive.
3. Merge overlapping user-facing docs into one product path.
4. Turn script docs into real .ps1 files stored in GitHub as canonical source.
5. Build a v1 product folder for download.


SAFE RULE
Do not keep every doc live.
Keep only the best canonical versions live.
Everything else moves to Build or Archive.


OUTCOME
Mum PC Helper becomes:
- tidy
- product-shaped
- easier to maintain
- easier to download
- easier to clone for future support products
